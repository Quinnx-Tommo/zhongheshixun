# 2026-07-07-核心功能取舍.md

## 决策：3 个 docx 核心功能保持 P2 状态，不加入 MVP 设计

### 背景
tech-director 对照 docx 原始需求（`11.基层卫生人员网络培训平台.docx`）审查设计，发现 3 个 docx **明确写出的功能**在设计中缺失：

| 功能 | docx 原文 | 当前状态 |
|------|---------|---------|
| 对培训师提问 | "可以支持对该课程的培训师进行提问" | P2 可选 |
| 学习笔记 | "可以记录该课程的学习笔记" | P2 可选 |
| 课程评分评论 | "学员可以对课程进行评分评论" | P2 可选 |

### 决策
**保持现状，不把这 3 个功能加入设计。**

### 决策理由
1. 需求偏差说明 v2.0 已经把这 3 项标为 P2 可选（§4.1 表第 8/9/10 行、§七）
2. 毕设 MVP 聚焦核心链路（学习 / 考试 / 管理），这 3 项是"锦上添花"
3. 时间约束（1-2 周开发周期），优先保证主线质量
4. 答辩时可用话术："期末作业时间约束，学习笔记、课程评分、对培训师提问作为 P2 功能预留，0.5-1 天/项即可补充"

### 后续影响
- `docs/database.sql` 不加新表
- `docs/dev-api.md` 不加新接口
- `docs/dev-miniapp.md` 不加新页面
- `需求偏差说明.md` 保持 v2.0 不变

### 后续可补（时间允许时）
- qa_question 表 + API（对培训师提问）
- study_note 表 + API（学习笔记）
- course_review 表 + API（课程评分评论）
