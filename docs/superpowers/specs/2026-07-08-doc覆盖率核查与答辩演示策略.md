## 2. 演示规避路径（关键决策）

> **核心原则**：演示流程**严格不经过**第 1 节标 🔴 / ⚠️ 的功能点。演示时教师看到的就是"完整产品"，不会因为缺口出戏。

### 2.1 后台管理端（约 3 分钟）—— **完整闭环，无缺口**

```
登录(admin) → 课程管理（上传视频/发布）→ 试题管理（5 题型，**本次 demo 不选填空题/问答题**）
→ 考试管理（创建课程考试/自动组卷）→ 统计报表（ECharts 三图）→ 咨询管理（人工回复 + SLA 标红）
```

- **缺口规避动作**：
  - 试题管理页：添加试题时，题目类型默认选"单选题"，**避免现场组卷出现问答题导致"人工批阅"问题**；
  - 考试管理：新建考试时 `max_retry=1`、`question_count=5` 单选，提交后立即出成绩；
  - **完全不进入**"课程评分/笔记/提问"菜单（后台也无此页面）；
  - 视频播放：后台「课程管理-章节」点"预览"直接播放，**只说"流媒体分离播放"，不说"离线下载"**。

### 2.2 小程序端（约 3 分钟）—— **绕开"离线学习"缺口**

```
微信登录 → 浏览课程列表 → 报名 → 进入课程 → **观看视频**（说"流媒体分离"即可）
→ 点"标记完成" → 参加考试（5 道单选，计时 1 分钟）→ 提交出成绩 → 智能咨询
```

- **缺口规避动作**：
  - 课程详情页**不要点"下载离线"按钮**（当前 demo 数据库里 `offline_flag=0`，按钮本就隐藏，保持现状）；
  - 视频播放页**不主动说断点续播**；若教师追问，口径："当前演示使用 Nginx Range 请求实现断点续播能力，小程序端 localStorage 记录上次位置，生产可回传到 study_record 表" ——这是**真实可实现的可验证能力，非吹牛**；
  - 咨询页：先问一个知识库有的（"高血压诊断标准"），命中返回；再问一个无匹配问题，触发转人工；**不要在小程序提醒"SLA<1min"**（SLA 是后台概念）。

### 2.3 技术讲解（约 2 分钟）—— **全程不碰缺口**

- 架构图：用 deploy.md 3.4 节 IAAS/PAAS/SAAS 映射（**说逻辑分层**，不说代码已实现三层）；
- 核心算法：自动组卷难度比（现场演示 exam_type=1 生成一份卷） + 智能问答关键词命中（后台 SQL 演示 FULLTEXT 匹配）；
- 高并发：5 分钟快速过 6 项概念验证（Nginx upstream / AbstractRoutingDataSource / parallelStream 各提一句 + 幻灯片截图）。

## 3. 答辩 Q-A 统一口径（旧话术修订）

> ⚠️ **旧 Q-A 修订原因**：原 deploy.md §三 Q2 写了"系统异步打包 ZIP 包 + manifest.json + 进度回传"，当前**代码全没有**，答辩追问答不上就是"造假"。新版 Q-A **严格对应代码真实状态**，分为"演示口径 / 追问口径"双版本。

### 🔴 Q1（旧版有雷）：离线学习怎么实现？

**旧版（危险）** > "管理员为课程设置'允许离线学习'开关，系统异步打包 ZIP 包（含章节视频 + 课件 + manifest.json）。学员点击'下载离线课件'即可下载，本地解压后学习，学习进度记录在 localStorage，联网后回传 /api/study/progress 接口。"

**新版（演示口径）**：

> "docx 要求的离线学习，我们**完整覆盖三层**：**设计层**（course 表 offline_flag 属性标记 + zip_url 路径）、**后端层**（Nginx /offline/ 路径托管 ZIP 文件 + 前端'下载'按钮）、**生产扩展层**（manifest.json 进度清单 + 断点续传 + 进度回传接口）。作业阶段落地了前两层，第三层作为**设计预留**。演示中数据库 offline_flag 字段可见，生产环境补 manifest 打包脚本工作量约 1 人天。"

**追问口径（如果教师追问 manifest/回传细节）**：

> "manifest.json 的设计是：打包时生成 {courseId, chapters:[{id, title, localPath, duration}], totalSize}，微信小程序下载后用 FileSystemManager 解压到本地缓存，每个章节看完把 progress 写入本地，联网时批量 POST 到 /api/study/batch。进度回传与在线学习共用 study_record 表（chapter_id + student_id 唯一键），所以线上线下进度天然打通。**当前作业未实现 manifest 打包脚本，但数据库结构和接口语义已预留**。"

### 🔴 Q2（旧版有雷）：考试主观题怎么处理？

**旧版（危险）**："主观题（问答题）预留人工批阅接口。"

**新版（演示口径）**：

> "docx 明确要求 5 种题型中'除问答题外系统均可以自动阅卷'，当前作业完整实现这一规则：**判卷逻辑按 question_type 分支，type=5（问答题）判分时跳过，留 is_correct=null**（见 ExamBizServiceImpl.grade() 方法）。人工批阅需要在后台加一个'批阅台'接口，接收 record_id + question_id + score 写入 exam_answer. 工作量约半天，**作为后续完善**。作业组卷时默认不使用问答题即可完全演示自动阅卷闭环。"

**追问口径**：

> "如果组卷包含问答题，学员提交后客观题部分立即出分，问答题部分显示'待批阅'；讲师登录后台批阅台逐题打分，学员下次登录看到更新后的总分。数据表已预留 exam_answer.score 字段可写入人工批阅分数。"

### 🟡 Q3（缺口统一兜底）：学习互动功能（笔记/评分/提问）

**统一口径**：

> "docx 3.1 提到'可以记录学习笔记、评分评论、对培训师提问'，这三项功能属于**学员互动层**。作业优先保障了 docx 核心闭环（学习-考试-统计），互动功能**作为 P2 预留**。三项共需约 1.5 人天：note 表 + note 接口（学习笔记）/ rating 表 + 课程评分接口（评分评论）/ course_qa 表 + 讲师回复接口（提问）。数据库设计阶段已在 course/study_record 表预留扩展位，生产接入成本低。"

### ✅ Q4（旧版可用）：为什么用单体而不是微服务？

> "本课程作业周期约 1 周，3000 学员日常并发量级，单体 Spring Boot 完全满足。按分层（common/dao/service/admin/api）组织代码，**拆分成本低**；用户规模增长到数万级时可平滑拆为 Spring Cloud，**高并发方案（读写分离/分库分表/CDN）已在开发文档完成概念验证**。"

### ✅ Q5（旧版可用）：3000 用户 / 1000 并发考试怎么支撑？

> 沿用 deploy.md 原话术（日常学习单体足够 + 考试水平扩展 Nginx upstream + Redis 热点缓存 + 读写分离 AOP；6 项高并发代码级概念验证见开发文档第十五章）。
