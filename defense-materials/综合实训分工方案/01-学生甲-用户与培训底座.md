# 学生甲 — 用户体系 + 培训底座（系统入口 + 治理）

> **综合实训角色**：学生甲  
> **业务域**：用户体系 + 培训底座  
> **核心叙事**：「系统怎么认人 → 人分哪几种 → 培训怎么组织」

---

## 一、业务定位

你是整个系统的**入口和底座**。你的模块回答三个问题：

1. **谁进来了？** — 登录认证（JWT 签发与校验）
2. **他是谁？** — 用户角色（管理员 / 讲师 / 学员 3 种 RBAC）
3. **培训怎么组织的？** — 培训计划编排（训练计划 → 关联课程 → 流程控制）

其他 3 位学生（乙/丙/丁）的前置依赖都是你：
- 乙的课程需要"学员"角色才能报名
- 丙的考试需要"学员"角色才能答题
- 丁的咨询需要"学员"角色才能提问

**响应**：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiJ9.xxxx",

    "username": "admin",
    "role": "ADMIN",

```

```

│ 前端 │          │ AuthController│          │ JwtTokenProvider│        │ 数据库     │
└──┬───┘          └──────┬───────┘          └──────┬─────────┘        └─────┬──────┘

   ├─────────────────────▶│                        │                        │
   │                      │ 1. 查用户              │                        │

   │                      │ 3. status=1 检查       │                        │

   │                      │                        │                        │
   │ ========== 后续请求带 token ==========          │                        │
   │                      │                        │                        │
   │ GET /api/xxx         │                        │                        │
   │ Authorization: Bearer xxx                     │                        │
   ├────────────────────────────────────────────────────────────────────────▶│
   │                      │                        │                        │

   │                      │ 7. 注入 userId/role   │                        │
   │                      │    到 SecurityContext │                        │
   │                      │ 8. 放行 → Controller  │                        │
```

### 4.2 用户管理接口（UserManagementController）

| 方法 | 路径 | 请求体 | 响应 | 说明 |
|------|------|--------|------|------|
| GET | `/admin/user/list?page=1&size=10&role=STUDENT` | — | `{list, total}` | 分页 + 按角色筛选 |
| GET | `/admin/user/detail/{id}` | — | `{user}` | 用户详情（不含 password） |
| POST | `/admin/user/add` | `{username, password, realName, role, ...}` | — | 密码 BCrypt 加密入库 |
| PUT | `/admin/user/edit/{id}` | `{realName, phone, ...}` | — | 编辑（不直接改密码） |
| PUT | `/admin/user/status/{id}` | `{status: 0/1}` | — | 启用/禁用 |
| DELETE | `/admin/user/delete/{id}` | — | — | 逻辑删除 |

**关键实现**：
- `add` 时密码必须 `BCryptPasswordEncoder.encode(明文)` 后再入库
- `list` 返回前过滤 `password` 字段（安全）
- `delete` 改 `deleted=1`，不真删

## 五、综合实训深度点（3 个）

### 5.1 JWT 认证全流程（核心亮点，必须能画时序图）

**代码位置**：`JwtTokenProvider.java` + `AuthController.java` + `JwtAuthenticationFilter.java`

**关键代码**：

```java
// JwtTokenProvider.java - 生成 token
public String generateToken(SysUser user) {
    return Jwts.builder()
        .setSubject(user.getId().toString())
        .claim("username", user.getUsername())
        .claim("role", user.getRole())
        .claim("realName", user.getRealName())
        .setIssuedAt(new Date())
        .setExpiration(new Date(System.currentTimeMillis() + 7200_000)) // 2h
        .signWith(SignatureAlgorithm.HS256, secretKey) // secret=训练平台密钥
        .compact();
}

// JwtAuthenticationFilter.java - 验 token
String token = request.getHeader("Authorization").replace("Bearer ", "");
Claims claims = Jwts.parser().setSigningKey(secretKey).parseClaimsJws(token).getBody();
Long userId = Long.parseLong(claims.getSubject());
String role = claims("role").toString();
// 注入 SecurityContext
```

**教学点**：
- 为什么 JWT 比 Session？（无状态、分布式友好）
- secretKey 为什么必须 9898/9899 一样？（跨服务验 token）
- token 过期怎么办？（返回 401，前端跳登录）
- token 被盗怎么办？（HTTPS + 短有效期 + refresh token 预留）

### 5.2 3 角色 RBAC 权限映射

**角色定义**：

| 角色 | 代码标识 | 权限范围 |
|------|----------|----------|
| **管理员** | ADMIN | 全部功能 |
| **讲师** | TEACHER | 课程管理 + 试题管理 + 查看学员进度 + 答疑 |
| **学员** | STUDENT | 课程学习 + 考试 + 个人中心 |

**权限控制实现**：

```java
// Controller 层
@PreAuthorize("hasRole('ADMIN')")
@GetMapping("/admin/user/list")
public Result list() { ... }

// 或手动检查
if (!"ADMIN".equals(role)) {
    throw new BusinessException(ErrorCode.FORBIDDEN);
}
```

**教学点**：
- 为什么不用按钮级别权限？（作业简化，菜单级即可）
- 怎么防越权？（接口层校验 role，不能只靠前端隐藏按钮）

### 5.3 3 种培训模式 + 3 类考试

**培训模式**（docx 3.1 强制要求）：

| 模式 | 实现方式 | docx 原文 |
|------|----------|-----------|
| 公开课 | `course_type = 1` | 允许学员自主选择学习 |
| 必修课 | `course_type = 2` | 由管理人员分配，针对不同组织机构提供个性化、定制化课程 |
| 计划流程性培训 | `train_plan` + `plan_course.sort_order` | 课程和测评通过一定顺序关系和条件关系组合为完整的学习培训方案 |

**考试类型**（docx 3.2 强制要求）：

| exam_type | 考试类型 | docx 原文说明 |
|-----------|---------|-------------|
| 1 | 课程考试 | 课程关联，学完课程后参加，未通过可重考 |
| 2 | 计划内考试 | 作为计划里面的子活动，可作为流程通过条件或计划完成条件 |
| 3 | 单独考试 | 独立于课程/计划，一般可用于定期测评 |

## 九、常见 Bug 与解决

| 现象 | 原因 | 解决 |
|------|------|------|
| 登录失败 "密码错误" | 密码是 `123456` 不是 `password` | 改密码或改 DB hash |
| token 过期 401 | 2h 无操作 | 重新登录 |
| 按钮能看到但点完 403 | 前端菜单显示，后端 role 校验失败 | 接口层检查 role 是否匹配 |
| 添加用户失败 "用户名已存在" | username 唯一约束 | 换 username |
| 删除用户后还能登录 | 逻辑删真删？ | 确认是 `deleted=1` 不是真删 |
| 培训计划关联课程失败 | course_id 不存在 | 先建课程再关联 |
