## 学生甲模块概览

| 维度 | 内容 |
|------|------|
| **业务定位** | 系统入口与治理底座，提供统一身份认证 + 3 角色 RBAC + 培训计划编排 |
| **核心职责** | ①登录认证（JWT 签发/校验） ②3 角色 RBAC（admin/teacher/student） ③培训计划编排 |
| **关键代码** | `AuthController.java`、`UserManagementController.java`、`TeacherController.java`、`TrainPlanController.java`、`JwtTokenProvider.java`、`JwtAuthenticationFilter.java` |
| **数据库表** | `sys_user`、`teacher`、`train_plan`、`plan_course` |
| **依赖关系** | **不依赖** 乙/丙/丁（dashboard 会调用乙的 stats 接口展示概览） |
| **被依赖** | 乙 / 丙 / 丁 **全部** 依赖甲的 token（userId + role + realName） |
| **端口** | 9898（admin 端） |
| **数据库** | 1 个 MySQL 8.0 实例，4 张表 |

### 学生甲业务域速记

> **学生甲是「认证中心」**——所有跨模块的 token 都来自甲，甲不调用乙丙丁，但乙丙丁全部依赖甲。

### 4 张表结构速记

| 表名 | 核心字段 | 角色 |
|------|----------|------|
| `sys_user` | id, username, password(bcrypt), role, real_name, phone, deleted | admin / teacher / student 三角色共用 |
| `teacher` | id, user_id, name, title, hospital, specialty | 师资档案（被乙的 course 引用） |
| `train_plan` | id, name, year, region, created_by, deleted | 培训计划主表（年/地区） |
| `plan_course` | id, plan_id, course_id | 计划-课程关联（M:N） |

### 2.3 技术可行性与经济可行性分析方法

#### 2.3.1 技术可行性分析维度

| 维度 | 分析方法 | 结论 |
|------|----------|------|
| **JWT 签发与校验** | 统一密钥 + 3600s 过期 + refresh_token 机制 | ✔ Spring Security + jjwt 成熟方案 |
| **3 角色 RBAC** | admin / teacher / student 三角色，注解级权限控制 | ✔ @PreAuthorize + 自定义注解 |
| **培训计划 M:N** | 计划可包含多门课程，课程可属于多个计划 | ✔ plan_course 关联表 |
| **密码安全** | BCrypt cost=10 + 复杂度校验（8 位 + 大小写数字） | ✔ Spring Security 密码编码器 |
| **跨端 token 一致性** | 9898 / 9899 共用同一密钥，userId 一致 | ✔ 配置化 jwt.secret |

#### 2.3.2 经济可行性分析维度

| 维度 | 估算 | 说明 |
|------|------|------|
| **甲模块人力** | 1 人 × 2 月 × 8000 元 | 学生甲独立完成 |
| **认证库依赖成本** | jjwt / spring-security = 0 元 | Apache 2.0 / MIT |
| **第三方服务** | 0 元 | 无短信网关（注册仅限管理员录入） |
| **甲模块云资源** | 与 9898 端口共享 | 无独立费用 |
| **甲模块总成本** | ≈ 1.6 万元 | 占全项目 24% |

#### 2.3.3 多元解决方案提炼路径

> **方法论**：调研同类平台（好医生/华医网/CME）登录方案 → 提炼 3 套候选方案 → 选最适合基层医疗培训的方案。

- **方案 A：JWT 无状态 token（选定）**：适合多端、移动优先
- **方案 B：Session + Redis 共享**：适合纯 Web、不跨端
- **方案 C：OAuth 2.0 第三方授权**：适合对接微信/钉钉登录（本项目可预留）

| 序号 | 文献/报告 | 来源 | 核心观点 |
|------|-----------|------|----------|
| 1 | 《RBAC 模型在医疗信息系统中的设计与实现》 | 计算机应用研究 | 医疗信息系统建议采用 RBAC0/RBAC1 角色分层模型，配合 @PreAuthorize 注解实现细粒度权限 |

3. **培训计划选 3 维度**：年度+地区+课程，符合医学继续教育研究结论
4. **差异化优势**：4 个标杆**均无**培训计划编排能力 → 本项目（甲模块）填补空白

## 第四章 同类项目技术方案对比表

### 4.1 三种可替代认证方案

#### 方案 A：JWT 无状态 token（**甲最终选型**）

- **架构**：Spring Security + jjwt 0.11.5 + BCrypt
- **部署**：单机 9898 端口
- **适用规模**：3 000 - 50 000 用户
- **优势**：跨端、移动优先、无状态、性能高
- **劣势**：token 撤销需额外机制（黑名单）

#### 方案 B：Session + Redis 共享

- **架构**：Spring Session + Redis 7.x
- **部署**：单机 9898 端口 + Redis 实例
- **适用规模**：3 000 - 30 000 用户
- **优势**：撤销简单、Spring 原生支持
- **劣势**：跨域共享复杂、不适合移动端

#### 方案 C：OAuth 2.0 第三方授权

- **架构**：Spring Authorization Server + 微信开放平台
- **部署**：独立授权服务器
- **适用规模**：> 10 000 用户
- **优势**：对接微信/钉钉体验最佳
- **劣势**：开发成本高、依赖第三方

### 4.2 《甲模块同类技术方案对比表》

| 对比维度 | 方案 A：JWT（甲选型） | 方案 B：Session + Redis | 方案 C：OAuth 2.0 |
|----------|----------------------|------------------------|------------------|
| **架构成熟度** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **开发成本（人月）** | 1.0 - 1.5 | 1.5 - 2.0 | 3.0 - 5.0 |
| **跨端支持** | ✔ 完美 | ⚠ 需 CORS | ✔ 完美 |
| **移动端适配** | ✔ 完美 | ⚠ 需 cookie 共享 | ✔ 完美 |
| **token 撤销** | ⚠ 需黑名单 | ✔ 直接失效 | ✔ 撤销端点 |
| **可扩展性** | 微服务友好 | 中 | 极高 |
| **性能** | < 5ms 校验 | 10-20ms（含 Redis） | < 10ms |
| **第三方依赖** | 无 | Redis 强依赖 | 微信/钉钉 |
| **毕设答辩适配** | ✔ 主流、可讲解 | ✔ 主流、可讲解 | ⚠ 偏复杂 |
| **代表案例** | Spring 全家桶默认 | 传统电商 | 微信生态 |

### 4.3 选型决策（甲）

> **结论**：选 **方案 A（JWT）**，理由：
> 1. 跨 admin / api / web / 小程序 4 端，无状态 token 是最简方案
> 2. 1.0 - 1.5 人月开发，1 人 2 周可完成
> 3. 答辩时可讲解 JWT 原理（已纳入毕设论文）
> 4. 预留方案 C 演进路径（小程序端可对接微信授权）
> 5. 数据库已采用 BCrypt cost=10（4 张表已设计 `password` 字段）

## 第五章 技术可行性分析

### 5.1 功能可实现性

| 功能模块 | 实现技术 | 关键挑战 | 解决方案 |
|----------|----------|----------|----------|
| 用户登录 | JWT + BCrypt | 跨端 token 一致性 | 统一密钥（jwt.secret 配置） |
| 3 角色 RBAC | @PreAuthorize | 注解粒度 | 角色-资源-操作 三级映射 |
| 培训计划编排 | plan_course 关联表 | 课程-计划 M:N | 中间表 + 事务保证 |
| 密码安全 | BCrypt cost=10 | 弱口令 | 8 位 + 大小写数字校验 |
| 跨端 token 互通 | 9898/9899 共密钥 | 配置漂移 | 配置中心统一管理 |
| 培训计划下发 | plan API | 与丁小程序联动 | 提供 planApiController（9899） |
| 用户管理（增删改） | UserManagement | 软删除 | `deleted=0/1` 标记 |

### 5.2 性能可达成性

- **登录 QPS**：单机 1000+ QPS（BCrypt 100ms × 1 线程 = 10 并发；线程池 20 = 200 并发）
- **培训计划查询**：分页 + 索引（plan_id / year / region）→ 1000 QPS 富余
- **token 校验**：纯内存运算，< 5ms
- **RBAC 权限校验**：注解级 AOP 拦截，< 2ms

### 5.3 技术风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| JWT 密钥泄露 | 中 | 配置中心 + 90 天轮换 + 定期审计 |
| 弱口令 | 中 | 注册时强制 8 位+复杂度校验 |
| 越权访问 | 中 | @PreAuthorize 注解 + 集成测试覆盖 |
| 培训计划数据丢失 | 中 | 每日 MySQL 全量备份 + binlog |
| JWT token 撤销 | 低 | 短过期（3600s）+ 黑名单（Redis） |

## 第七章 数据安全初步规划

### 7.1 甲模块数据分级

| 数据级别 | 类别 | 保护策略 |
|----------|------|----------|
| **L4 高敏感** | password、phone、id_card | BCrypt 哈希 + 脱敏 |
| **L3 中敏感** | role、real_name、登录日志 | 加密传输 + 权限隔离 |
| **L2 低敏感** | username、region | 访问控制 |

### 7.2 安全防护体系（甲模块）

```
┌──────────────────────────────────────────────┐
│              认证安全层（甲）                  │
│  • JWT 鉴权 + Spring Security 路由守卫         │
│  • BCrypt cost=10 密码哈希                     │
│  • @PreAuthorize 注解级 RBAC                  │
│  • SQL 注入防护（MyBatis-Plus 参数化）         │
└──────────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────────┐
│              传输安全层                        │
│  • HTTPS（Let's Encrypt）                     │
│  • JWT 不在 URL 传递（仅 Header）              │
│  • 敏感字段前端脱敏（手机号 138****0000）        │
└──────────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────────┐
│              存储安全层                        │
│  • 密码 BCrypt（不存明文）                      │
│  • 身份证 AES-256（预留字段）                  │
│  • 登录日志保留 90 天                           │
└──────────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────────┐
│              运维安全层                        │
│  • jwt.secret 配置中心管理                      │
│  • 90 天密钥轮换                               │
│  • 登录失败 5 次锁定 30 分钟                    │
└──────────────────────────────────────────────┘
```

### 7.3 合规对照

| 法规 | 要求 | 甲模块落实 |
|------|------|------------|
| 《网络安全法》 | 数据分类分级保护 | ✔ §7.1 |
| 《个人信息保护法》 | 知情同意、最小必要 | ✔ 注册协议 + 字段脱敏 |
| 《数据安全法》 | 风险评估、应急响应 | ✔ 备份 + 审计 |

### 7.4 数据安全风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| JWT 密钥泄露 | 中 | 配置中心 + 90 天轮换 |
| 暴力破解 | 中 | 5 次失败锁定 30 分钟 |
| 弱口令 | 中 | 8 位+复杂度校验 |
| 越权访问 | 中 | @PreAuthorize 注解 |
| 培训计划数据丢失 | 中 | 每日 MySQL 全量备份 |
