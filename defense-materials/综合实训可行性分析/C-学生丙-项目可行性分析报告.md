### 2.3 技术可行性与经济可行性分析方法

#### 2.3.1 技术可行性分析维度

| 维度 | 分析方法 | 结论 |
|------|----------|------|
| **自动组卷算法** | 难度比 30:50:20 + 随机抽取 + 知识覆盖 | ✔ Java Stream + Random 实现 |
| **自动阅卷算法** | 单选/判断：直接匹配；多选：排序后比较；简答：人工 | ✔ 多选排序是经典算法 |
| **5 题型 + 3 难度** | 题型枚举 5 种（单选/多选/判断/填空/简答），难度 1-3 | ✔ 字段枚举 |

| **ECharts 三图** | 柱状/折线/饼图 + 复杂查询 | ✔ Vue 组件化 |

#### 2.3.2 经济可行性分析维度

| 维度 | 估算 | 说明 |
|------|------|------|
| **丙模块人力** | 1 人 × 2 月 × 8000 元 | 学生丙独立完成 |
| **算法库依赖** | ECharts Apache 2.0 免费 | 0 元 |

#### 2.3.3 多元解决方案提炼路径

> **方法论**：从同类平台考试系统调研 → 提炼 3 套组卷方案 + 3 套阅卷方案 → 选最适合基层医疗的方案。

- **组卷方案 A：固定模板抽题（选定）**：简单可解释，符合毕设答辩
- **组卷方案 B：遗传算法寻优**：偏研究性，不适合毕设
- **组卷方案 C：人工选题**：传统方式，不智能
- **阅卷方案 A：客观题自动 + 主观题人工（选定）**：平衡效率与质量
- **阅卷方案 B：AI 阅卷**：依赖第三方 API
- **阅卷方案 C：纯人工**：效率低

## 第四章 同类项目技术方案对比表

### 4.1 三种可替代组卷方案

#### 方案 A：固定模板抽题（**丙最终选型**）

- **架构**：难度比 30:50:20 + 题型数量 + 知识点覆盖
- **实现**：Java Stream + Random 随机抽取
- **优势**：实现简单、可解释、答辩演示清晰、性能高
- **劣势**：组卷多样性略低于遗传算法

#### 方案 B：遗传算法寻优

- **架构**：染色体编码 + 适应度函数 + 交叉变异
- **优势**：组卷质量高（论文级别）
- **劣势**：实现复杂、答辩不友好、性能开销大

#### 方案 C：人工选题

- **架构**：管理员从题库手动选 N 道题
- **优势**：完全可控
- **劣势**：效率低、不可重复

### 4.2 三种可替代阅卷方案

#### 方案 A：客观题自动 + 主观题人工（**丙最终选型**）

- **实现**：单选/判断/填空直接匹配；多选排序后字符串比较；简答人工
- **优势**：平衡效率与质量、算法可讲解

#### 方案 B：AI 阅卷

- **实现**：调用百度/讯飞 API
- **优势**：可处理主观题
- **劣势**：成本高、依赖第三方

#### 方案 C：纯人工阅卷

- **优势**：质量高
- **劣势**：效率低、毕设答辩不演示

### 4.3 《丙模块同类技术方案对比表》

| 对比维度 | 方案 A：固定模板抽题（丙选型） | 方案 B：遗传算法 | 方案 C：人工选题 |
|----------|----------------------------|------------------|------------------|
| **组卷质量** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **实现复杂度** | ⭐⭐ 低 | ⭐⭐⭐⭐⭐ 极高 | ⭐⭐ 低 |
| **答辩可讲解** | ✔ 清晰 | ⚠ 偏研究 | ✔ 清晰 |
| **性能** | < 100ms | 1-5s | 人工秒级 |
| **多样性** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **可重复性** | ✔ 种子可控 | ⚠ 随机性大 | ✔ 完全可控 |
| **代表案例** | 大多数考试系统 | 部分论文原型 | 传统题库 |

| 阅卷方案 | 方案 A：客观自动+主观人工（丙选型） | 方案 B：AI 阅卷 | 方案 C：纯人工 |
|----------|--------------------------------------|-----------------|----------------|
| **效率** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |
| **质量** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **成本** | 低 | 中（API 费） | 高（人力） |
| **答辩可讲解** | ✔ 算法清晰 | ⚠ 依赖第三方 | ✔ 但演示慢 |

### 4.4 选型决策（丙）

> **结论**：
> - **组卷选方案 A**（固定模板抽题 30:50:20）：实现简单、可讲解、满足毕设
> - **阅卷选方案 A**（客观自动 + 多选排序比较 + 主观人工）：平衡效率与质量

理由：
1. 答辩时需现场演示自动组卷 + 自动阅卷 → 方案 A 1s 内可完成
2. 4 个标杆均采用类似方案（客观自动 + 主观人工）
3. 多选排序比较比 Set 比较更稳健（文献支撑）
4. 7 统计接口 + ECharts 满足多角度统计要求
5. 数据库已采用该结构（6 张表已设计）

## 第五章 技术可行性分析

### 5.1 功能可实现性

| 功能模块 | 实现技术 | 关键挑战 | 解决方案 |
|----------|----------|----------|----------|
| 题库管理 | MyBatis-Plus | options JSON 字段 | Jackson 序列化 |
| **自动组卷** | Java Stream + Random | 难度比保证 | 3 段式筛选 + 随机 |
| 考试发布 | exam + paper 关联 | 题目数量校验 | 事务保证 |
| **双保险计时** | 前端 + @Scheduled | 超时自动交卷 | 后端兜底 |
| **自动阅卷** | 字符串匹配 + 排序 | 多选漏选/多选 | 排序后比较 |
| 成绩单 | exam_record + exam_answer | 复杂聚合 | SQL JOIN |
| 7 统计接口 | SQL 聚合 + Redis | 实时性 | 缓存 1 分钟 |
| ECharts 三图 | Vue 组件化 | 复杂查询 | 7 统计接口支撑 |

### 5.2 性能可达成性

- **自动组卷**：单次 < 100ms（Java Stream + 索引）
- **自动阅卷**：单次 < 500ms（多选排序比较）
- **考试并发**：1000 并发 < 2s 响应
- **7 统计接口**：缓存命中 < 50ms，未命中 < 500ms

### 5.3 技术风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 组卷多样性不足 | 低 | Random 种子可控 + 知识点覆盖 |

## 第七章 数据安全初步规划

### 7.1 丙模块数据分级

| 数据级别 | 类别 | 保护策略 |

| **L2 低敏感** | 题库内容、试卷 | 访问控制 |
| **L1 公开** | 考试名称、时长 | 公开 |

### 7.2 安全防护体系（丙模块）

```
│  • @PreAuthorize 注解级权限                    │
└──────────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────────┐
│              传输安全层                        │
│  • HTTPS 考试页                              │

└──────────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────────┐
│              存储安全层                        │
│  • 答案字段加密存储                            │
│  • 阅卷后清除 user_answer 明文                 │
│  • 考试记录 5 年保留（医学合规）                 │
└──────────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────────┐
│              运维安全层                        │
│  • 阅卷日志审计                               │
│  • 题目变更记录                                │
│  • 异常考试监控                                │
└──────────────────────────────────────────────┘
```
