## 一、项目概述

### 1.1 编写目的

本文档定义"四川省基层卫生人员网络培训平台"的 **完整功能需求、非功能需求、
验收准则、MoSCoW 优先排序与需求跟踪矩阵（RTM）**，用于：

1. 界定系统的软件功能边界与质量属性；
2. 作为概要设计、数据库设计、编码实现与测试用例的输入依据；
3. 作为教师评审和答辩验收的正式需求基线；
4. 在项目全生命周期中体现 RTM 的持续可追踪性。

### 1.2 项目背景

四川省 67 个民族县的 **基层卫生技术人员**（含村医、乡镇卫生院 / 社区卫生服务中心人员、
公共卫生、医技等）约 **3000 人**，每人每年需完成至少 **200 学时** 的继续教育培训。

当前痛点包括：

- 地域分散、师资不足、集中培训成本高；
- 民族地区基层网络弱、离线与弱网场景普遍；
- 手工管理报名 / 考试 / 统计效率低、质量难追溯。

本文档据此提出一套 **在线 + 离线混合** 的培训方案，
功能覆盖课程、考试、咨询、培训计划与统计报表。

### 1.3 适用范围

本文档适用于：

- 小程序（学员端）；
- PC Web 学员端（`web-student`）；
- 后台管理端（Vue3 + Element Plus）；
- 后端服务 `training-admin :9898` 与 `training-api :9899`。

## 二、总体描述

### 2.1 系统边界

**系统内**：用户与权限、课程、章节与资源、学习、考试、题库、咨询、
知识库、培训计划、统计报表 8 大业务模块。

**系统外**：

- 微信小程序开放平台（授权码、头像与昵称、openId）；
- CDN（静态资源 / 大文件分发，生产方案）；
- MySQL 8.0（UTF8MB4）、Redis（7.x，可选）；
- Nginx（静态视频分发、反向代理）。

### 2.2 用户角色（RBAC）

| 角色 | 编码 | 描述 | 主要操作 |
| --- | --- | --- | --- |
| 系统管理员 | `ADMIN` | 平台治理 | 课程 / 题目 / 考试 / 用户 / 培训计划 / 统计 |
| 培训师 / 教师 | `TEACHER` | 院 / 校师资 | 课程与章节管理、答疑、人工咨询回复 |
| 学员 | `STUDENT` | 基层卫生人员 | 登录、在线 / 离线学习、考试、智能问答、个人中心 |

> 方案依据：RBAC 是教育、医疗信息系统的通用且主流的访问控制模型，
> 文献 [1] 在医学教育平台中采用 RBAC 结合 JWT 的混合认证方案，
> 可满足多角色细粒度权限场景。

### 2.3 运行环境

| 层次 | 配置 |
| --- | --- |
| 后端 | JDK 17 + Spring Boot 2.7 + MyBatis-Plus + MySQL 8.0 |
| 前台（管理） | Node 18 + Vue 3 + Element Plus |
| 前台（PC 学员端） | Vite + Vue 3 + TypeScript + Element Plus |
| 移动端 | 微信小程序（原生 WXML / WXSS / JS） |
| 运行系统 | Windows 11（开发）/ Linux（生产演示） |

### 2.4 设计约束

- 单体 Spring Boot，非微服务，满足毕业设计演示合理简洁；
- 三端共享同一数据库；密码统一 BCrypt + 统一盐值；
- 离线 / 弱网能力与少数民族语言预留作为 P2 扩展点。

### 4.2 系统安全需求（System Security）

| ID | 需求 | 方案 / 实现位置 |
| --- | --- | --- |
| NFR-SEC-01 | **身份认证**：三端统一 BCrypt + JWT（HS256 / RS256），统一拦截器 | `training-common` |
| NFR-SEC-02 | **会话安全**：JWT 过期时间 ≤ 24h，单点登出采用黑名单（Redis） | `AuthInterceptor` |
| NFR-SEC-03 | **密码强度**：长度 ≥ 8、必须含大小写 + 数字 | 前端 + 后端双层校验 |
| NFR-SEC-04 | **RBAC**：请求级权限码（`auth:read`、`user:write`、`stats:read`…） | 注解 + 拦截器 |
| NFR-SEC-05 | **HTTPS 传输**：生产环境强制 HTTPS，禁用不安全 TLS | nginx / Spring Boot 配置 |
| NFR-SEC-06 | **SQL 注入**：100% 参数化查询（MyBatis-Plus + #{}） | 全数据层 |
| NFR-SEC-07 | **XSS**：富文本 + 输出 HTML 转义 | 前端 + 响应过滤器 |
| NFR-SEC-08 | **CSRF**：同源策略 + 自定义头（JWT 存 localStorage） | 前端 axios 配置 |
| NFR-SEC-09 | **限流**：单 IP + 单用户双令牌桶限流 | AOP + Redis（可选） |
| NFR-SEC-10 | **安全审计**：登录 / 敏感操作 / 数据修改日志 | `sys_log` 表 |

> 方案依据：多层纵深防御（认证 → 权限 → 传输 → 输入 → 审计）是
> 医疗信息系统的通用安全基线 [4]。

### 4.3 数据安全需求（Data Security）

| ID | 需求 | 方案 |
| --- | --- | --- |
| NFR-DATA-01 | **密码**：BCrypt 哈希 + 公共盐（本平台 cost=10）| `PasswordEncoder` |
| NFR-DATA-02 | **敏感字段脱敏**：身份证、手机号在列表 / 日志中脱敏 | DTO 序列化 |
| NFR-DATA-03 | **医疗数据分级**：课程 / 试题 / 学员信息三级（内部 / 敏感 / 高敏） | 数据分级策略（`docs/dev-api.md`） |
| NFR-DATA-04 | **传输加密**：TLS 1.2+ 全链路 | nginx / Spring Boot |
| NFR-DATA-05 | **备份恢复**：每日 02:00 全量备份 + 30 天保留 | `mysqldump` 脚本 |
| NFR-DATA-06 | **软删除**：所有业务表统一 `deleted` 字段 | MyBatis-Plus |
| NFR-DATA-07 | **变更审计**：考试时间、人工评分留痕 | `*_log` 表 |
| NFR-DATA-08 | **数据生命周期**：注销用户彻底假名化 / 归档 | 业务规则 |

> 依据：GB/T 22239-2019 信息安全等级保护基本要求；
